CMIO

The CMIO (Common Music Input/Output) window is a graphical tool for generating musical event data to and from sound synthesis and display applications. The intial contents of the window's various tabbed pages can be preset by passing field values to the cmio function when the window is created. Since the target pages and the window's Systems and Files pages allows these systems be dynamically configured and loaded into Common Music.

Window Layout

The CMIO window is divided into four main components:

  • A Sources frame holds tabbed pages that relate to creating and generating event data.
  • A Targets frame holds tab pages related to sound sysnthesis and display configurations.
  • An Executive frame manages the system's working directory and current input/output file.
  • A message line displays information related to the ongoing tasks executed inside the window.

Each tabbed page contains a set of fields representing the input/output features of a given application. The right-justified pages relate to more general features in the window or Lisp environment.

Colorized Fields

A few coloration rules provide additional information about how fields on a page work:
  • Blue labels mark required fields that must contain infomation before the page can execute.
  • Yellow backgrounds indicate fields that will be evaluated when the information is accessed.
  • White backgrounds indicate fields that should contain plain text or are read to produce a Lisp value. It is possible to force evaluation in a white field by using the special #.expr notation, where expr is an expression to be evaluated for the contents of the field.
  • The Message line uses syntax highlighting to classify the type of message displayed: green represents a normal message or sucessful completion of a command, yellow indicates a warning or notification, red signifies an error condition.

Error handling and notification

A CMIO window simply facilitates tasks that would otherwise be accomplished by evaluating expressinos in the Lisp interpreter. As such, its actions and data specification are all accplished using the normal Lisp. However, the windnow itself accomplishes its task via GTK callbacks which involves the (foreign) execution of Lisp code. Generating Lisp errors under foreign callbacks is generally a bad thing because the Lisp process may become unresponsive or simply terminate execution without allowing the user to save current state.
