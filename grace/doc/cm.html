Introduction

The Common Music 3 branch is implemented in Chicken Scheme and optimized for realtime scheduling within Grace. Since Scheme and realtime scheduling have caused significant design changes to the underlying code -- and thereby making source compatability with earlier versions of CM not even a possibility -- I have taken the opportunity to add, remove and modify CM's API to make it leaner, more interactive and to minimize the necessity of "boilerplate" code to process lists of data.

The distinguishing features of Common Music 3 compared to the previous (CM 2) line:

  • Does not compile or load sources at runtime.
  • Does not use an object system or generic functions.
  • Simplifed frequency tools combine support for key, note, hz and pitch class manipuation. Note names can include quarter-tone inflections (< and >).
  • Terser function calling syntax to assist with interactive coding:
    • No dependance on quoted symbols or keywords in argument data.
    • Numerous functions expanded to handle lists as well as atomic data. This cuts down on looping boilerplate and reduces the number of functions in the API. In the documentation separate formals are provided in cases where list handling causes a significant difference in argument syntax.
    • Opt/key parameters with DSSSL style keywords (symbols ending with a colon) replace &optional and &keyword parameters.
    • Shorter names in general.
  • SAL and Scheme equivalence. Scheme formals are only shown when they differ in some significant way from the SAL versions.
  • Ports replace files.

Mapping and Transformation

note (freq) Function
→ string or list

Returns the note (string) of freq, which can be a key, note or list of the same. A note name consists of a pitch class letter (c d e f g a b), an optional sharp or flat letter (s f) an optional quarter tone inflection (< or >) and an octave number (00 to 9). Note names can be specified as strings, symbols or keywords. Octave numbers inside note lists can be omitted, in which case the current note's octave defaults to the last octave specified, or to octave 4.

key (freq) Function
→ number or list

Returns the key (number) of freq, which can be a note, hertz value or list of the same. If a key contains a fractional part it is interpreted as cents above the integer key number, e.g. 60.5 is 50 cents above Middle C.

hz (freq) Function
→ number or list

Returns the hertz value of freq, which can be a note, key or list of the same.

pc (freq) Function
→ number or list

Returns pitch class (0-11) of freq, which can be a note, key or list of the same.

transpose (freq, amount) Function
→ number or list

Transposes freq by amount. If freq and amount are both pitch classes then the transposition occurs in pitch class (mod 12) space. Otherwise freq should be a key, note or list of the same and it is transposed by amount.

invert (freq) Function
→ number or list

If freq is a pitch class or list of pitch classes the pitch class complement(s) are returned. Otherwise freq should be a list of notes or keys and the (strict) invertions around the first key or note in the list is returned.

scale (num, start, step...) Function
→ list

Returns a list of num keys starting on start and incremented by each step (positive or negative) read mod the number of steps. For example, (steps 24 60 2 1 ) would return an ascending 3 octave octatonic scale starting on key number 60.

rescale (x, x1, x2, y1, y2, [base 1]) Function
→ number or list

Returns a floating point value between y1 and y2 that is proportional to a floating point x or list of the same lying between x1 and x2. If base is 1 the rescaling is linear otherwise its exponential where base > 1 produce increasing spread as x approaches x2 and values < 1 produce decreasing spread.

discrete (x, x1, x2, list) Function

Patterns

Patterns are objects that generate data in different orderings. Patterns can used individually or nested to form local orderings (sub-patterns) within the surrounding pattern.

PatternDescription
Cycle generates its elements in a continuous loop
Linegenerates its elements sequentially and sticks on last element
Palindromegenerates its elements forwards and backwards
Heapgenerates random permutations of its elements
Rotationgenerates systematic permutations of its elements
Weightinggenerates elements from a weighted random distribution
Markovgenerates nth order Markov chains of its elements
Graphgenerates elements from a directed graph
make-cycle (items, for = #f, repeat = #f) Function
→ pattern

Returns a pattern that generates its elements in a continual loop. Items is the list of data to generate and can include sub-patterns. For sets the number of internal reads that will make up the next period, or chunk, of data returned from the pattern. This value can be a number or pattern of numbers and defaults to the length of items. Repeat sets an optional limit on the number of periods that the pattern can generate; once this limit has been reached the pattern will return an end of data marker instead of elements.

make-line (items, for = #f, repeat = #f) Function
→ pattern

Returns a pattern that generates items sequentially and sticks on the last item.

make-palindrome (items, for = #f, repeat = #f, elide = #f) Function
→ pattern

Returns a pattern that generates items forwards and backwards. Elide determines what happens to the first and last items when the pattern reverses itself. If the value is false then these items are repeated. If the value is true then neither item is directly repeated. Otherwise the value can should be a two element list {bool bool} specifying the repeat value for the first and last item.

make-weightings (items, for = #f, repeat = #f) Function
→ pattern

Returns a weighting pattern that generates items from a weighted distribution. Each item can be an item, subpattern, or list:

{item [weight 1] [min 1] [max #f]}

where item is the thing to generate, weight is its weight in the distribution relative to the other items in the pattern, min is the minimum number of times that item must be directly selelected before another item can be considered, and max is the maximum number of times that item can be directly selected before another item must be selected.

make-rotation (items, for = #f, repeat = #f, rotate = {0}) Function
→ pattern

Returns a pattern that rotates its items according to rotate, which can be a list or a pattern of lists of the format:

{start [end] [step 1] [width 1]}

make-markov (items, for = #f, repeat = #f, past = {...}) Function
→ pattern

Returns a pattern that generates its items according to (nth order) transitions lists:

{past ... -> {next weight} ...}

Items appearing before -> in the rule are the past choice(s) that trigger the transition and items following -> make up the weighted distribution determining the transition's next outcome. If past it supplied it will be the initial past choices to match the transitions against, otherwise the first transition rule is used.

make-graph (items, for = #f, repeat = #f) Function
→ pattern

Returns a pattern that generates its items from a directed graph of nodes with transition patterns. Each node is a list:

{item to [id]}

where item is the item or pattern to generate, to is an id or pattern of ids identifiying which node should follow the current node in the graph, and id is a unique identifier for the node which defaults to the node's position in items starting from 1.

next (pattern, [chunk #f]) Function
→ element or list

Returns one or more elements from pattern according to chunk. If chunk is false then the next element from pattern is returned. If chunk is a number then that many elements are read from the pattern and returned in a list. Otherwise if length is true then the next period's worth of data is read from the pattern and returned in a list.

eop? (pattern) Function
→ element

Returns true if pattern is at the end of its current period. Pattern can be the state returned from the pattern or the pattern itself.

eod? (pattern) Function
→ element

Returns true if pattern is at the end of its data. This condition can only happen if the pattern was created with a repeat value explicitly provided. Pattern can be the state returned from the pattern or the pattern itself.

Csound Port Methods

send "cs:i" ins, start, duration ...
send "cs:i" list

Sends instrument data to the open Csound port. The first value should be the instrument's number followed by the rest of its pfield values. For realtime audio the start value (pfield 2) should be 0. The "cs:i" message also accepts a single list of values to be specified, in which case the list should contain the pfield data to send to Csound. The "cs:i" message is also implemented as a Scheme function of the same name that can be called directly. For example, the following Scheme expressions are equivalent: 

(send "cs:i" 1 0 1 60 1000)
(cs:i 1 0 1 60 1000)
(define mypars '(1 0 1 60 1000))
(send "cs:i" mypars)
(cs:i mypars)

send "cs:f", num...
send "cs:f", list

Sends function table data to Csound. Similar to "cs:i" in all other respects.

send "cs:clear"

Clears the current contents of the port's score buffer.

The file "grace.orc" in the Grace resource directory can be used to experiment with the Csound Port. The file contains two sample instruments: i1 is a karplus-strong pluck sound and i2 is a sinewave. Both instruments take five identical parameters: ins beg dur keynum amp where ins is the instrument number (1 or 2), beg is the begin time of the sound in seconds (normally 0 for realtime audio), dur is the sound duration in seconds, keynum is the note's frequency specified as a MIDI key number, and amp is the note's amplitude (0-10000).

See Example 5 for an example of sending data to the Csound port.

Musical Processes

A musical process is a Scheme function of one argument that you sprout into the scheduler with an (optional) delay time. At the appropriate (real) time the scheduler calls the function and passes it the elapsed time since the process was sprouted. The function executes and returns either (1) the amount of time the scheduler should wait until it calls the function again or (2) a negative number if the function should not be called again.

The go Macro

The go macro is an easy way to create a process function, it has the same syntax as Scheme's do but returns an iterative process instead of performing iteration directly. It also wraps your code inside an error handler so Scheme can't crash from errors under the callback:

go ((var init [step]) ...) (stop? finally ...) body ...
After the name of the macro comes its binding list, each binding is a list containing the name of a local variable var followed by its initial value init and an (optional) stepping expression step. After the binding list comes the termination clause that is evaluated each time through the loop; if the stop? expression is true then any finally expressions are evaluated and the process stops running. The Scheduler

The scheduler is a JUCE thread that executes on a 1 millisecond tick. There are just a few entry points:

(sprout process &optkey (ahead 0) (id #f)) Scheme
sprout process, ahead = 0, id = #f SAL
→ void

Adds process to the scheduler starting at ahead time from now. If id is provided it tags the process in the scheduling queue; tagged processes can then be selectively removed from the scheduler using stop. The process parameter can be a single process or list of processes. If it is a list then ahead and id can also be lists and their values are paired with each process in left-to-right order. See Musical Processes for more information.

stop (id...) SAL
→ void

Stops processes currently running in the scheduling queue. If no ids are given all the processes are stopped, otherwise only processes with the specified ids are removed. Multiple processes can share a common id.

hush () SAL
→ void

Stops all processes that are currently running and flushes any pending events in the output queue.

pause () SAL
→ void

Suspends the process scheduler.

cont () SAL
→ void

Resumes (continues) the paused scheduler.

now () SAL
→ float

Returns the current time of the scheduler. See also elapsed

Function call notation

A function name followed by zero or more comma-delimited arguments enclosed within (). Several different types of parameters are supported:

Required parameters
Required parameters must be specified. A required parameter appers in the function definition documentation as parameter.
Optional parameters
Optional parameters may be specified or not. If not they receive a default value, which is false if not otherwise idncated. An optional parameter is shown in the function definition documentation as [parameter value]
Opt/key parameters
Opt/key parameters are optional parameters that can be specified positionally or by preceding the parameter value by the parameter's DSSSL keyword name (the name of the parameter with a colon at the end.) For example if a function foo has two opt/key parameters named a and b then it could be called in any of the following ways: (foo) (foo 4) (foo 2 3) (foo a: 3) (foo b: 99) (foo a: 2 b: 4) (foo b: 99 a: 2) An opt/key parameter is shown in the documentation as parameter = value
Variable parameters
Variable parameters may be specified zero or more times. A variable parameter is shown in the function definition documentation as parameter... where ... means "the rest of the arguments".

Example 1

Mapping pink noise onto scales.

Scheme

(define (pinkscale len scale)
  (go ((i 0 (+ i 1)))
      ((= i len) )
    (send "mp:midi" key: (discrete (ranpink) -1 1 scale))
    (wait .2)))

(define dorian (scale 50 20 2 1 2 2 2 1 2))

(sprout (pinkscale 100 dorian))

(let ((wierd (scale 40 20 2 (between 1 4) 2 (between 1 4) 2)))
  (sprout (pinkscale 100 wierd)))

SAL

define process pinkscale(len, scale)
  run repeat len
    send "mp:midi", key: discrete( ranpink(), -1, 1, scale)
    wait .2
  end

define variable dorian = scale(50, 20, 2, 1, 2, 2, 2, 1, 2)

sprout pinkscale(100, dorian)

begin with weird = scale(40, 20, between(1,4), 2, between(1,4), 2)
  print "scale=", weird
  sprout pinkscale(100, weird)
end

Example 2

Using ids to selectively stop processes.

Scheme

(define (chopin n r d k)
  (go ((i 0 (+ i 1)))
      ((= i n) #f)
    (send "mp:midi" dur: d key: k)
    (wait r)))

;; sprout D# minor chord with ids

(begin 
  (sprout (chopin 100 .5 .25 63) 0 1)
  (sprout (chopin 100 .5 .25 66) 0 2)
  (sprout (chopin 100 .5 .25 70) 0 3)
  (sprout (chopin 100 .5 .25 75) 0 1)
  )

;; first stop F#, then A# then the D# octave

(stop 2)
(stop 3)
(stop 1)

SAL

define process chopin (n, r, d, k)
  run repeat n
    send "mp:midi", dur: d, key: k
    wait r
  end

;; sprout D# minor chord with ids

begin
  sprout chopin(50, .5, .25, 63), 0, 1
  sprout chopin(50, .5, .25, 66), 0, 2
  sprout chopin(50, .5, .25, 70), 0, 3
  sprout chopin(50, .5, .25, 75), 0, 1
end

;; first stop F#, then A# then the D# octave

exec stop(2)
exec stop(3)
exec stop(1)

Example 3

Pentatonic wind chimes with amp envelope and exponential decay.

Scheme

(define (chimes n)
  (go ((i 0 (+ i 1))
       (l '(0 2 5 7 9 12))
       (k (+ 37 (* 12 (ran 6)))))
      ((= i n) #f)
    (send "mp:midi" 0 (between 2.0 4.0)
          (+ k (nth l (ran 6)))
          (interp i 0 90 n 30))
    (wait (rescale i 0 n .1 (between 1.5 2.0) 50))))

(sprout (chimes 20 ) )

SAL

define process chimes(n)
  run with l = {0 2 5 7 9 12}, k = 37 + (12 * ran(6))
    for i below n
    send "mp:midi", 0, between( 2.0, 4.0),
         k + nth( l, ran( 6)),
         interp(i, 0, 90, n, 30)
    wait rescale( i, 0, n, .1, between(1.5, 2.0), 50)
  end

sprout chimes( 20 )

Example 4

Triggering realtime processes via a MIDI input hook.

Scheme

;; define a process thats plays a major or minor scale based on a given key

(define (playscale key dur)
  (let ((scl (if (odds .5)
                 (scale 15 key 2 2 1 2 2 2 1)
                 (scale 15 key 2 1 2 2 2 1 2))))
    (go ((i 0 (+ i 1))
         (l (length scl))
         (r (/ dur (length scl))))
        ((= i l) #f)
      (send "mp:midi" key: (nth scl i) 
            dur: .5
            amp: (interp i 0 .9 l .5))
      (wait r))))

;; test the process
  
(sprout (playscale 60 1.2))

;; define a hook to play a scale based on a key you press...

(define (myhook m)
  (if (mm:on? m )
    (let ((k (mm:key m)))
      (sprout (playscale k 1.2)))))

;; set hook ...

(send "mp:inhook" myhook)

;; now play your MIDI keyboard ...  then clear your hook when done

(send "mp:inhook" #f)

SAL

define process playscale(key, dur)
  run with scl = odds(.5, scale(15, key, 2, 2, 1, 2, 2, 2, 1),
                      scale(15, key, 2, 1, 2, 2, 2, 1, 2)), 
           len = length(scale)
    for i below len
    send "mp:midi", key: nth( scl, i),
         dur: .5,
         amp: interp( i, 0, .9, len, .5)
    wait dur / len
  end

;; test the process
  
sprout playscale( 60, 1.2)

;; define a hook to play a scale based on a key you press...

define function myhook (m)
  if mm:on?(m) then
    begin with k = mm:key(m)
      sprout playscale(k, 1.2)
    end

;; set hook ...

send "mp:inhook", myhook

;; now play your MIDI keyboard ...  then clear your hook when done

send "mp:inhook", #f

Example 5

Sending audio data to the Csound port in realtime. (The i1 instrument is defined in the file 'grace.orc' located in the application's resource directory.)

Scheme

 
(define (rani1 len rhy lb ub amp)
  (go ((num 0 (+ num 1))
       (dur (* rhy 2))
       (key lb (ran lb ub)))
      ((= num len) #f)
    (send "cs:i" 1 0 dur key amp)
    (wait rhy)
    ))

(sprout (rani1 10 .2 60 72 1000))

SAL

define process ranins1(len, rhy, lb, ub, amp)
  run with dur = rhy * 2
    repeat len
    for key = lb then ran(lb,ub)
    send "cs:i", 1, 0, dur, key, amp
    wait rhy
  end

sprout ranins1(10, .2, 60, 72, 1000)
